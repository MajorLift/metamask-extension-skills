# Understand Before Acting

## When To Use

- Before making changes to any document, code, or artifact
- Before committing to an analysis approach
- When a request is ambiguous about document type (plan vs guide, spec vs tutorial)

## Pre-Edit Checklist

Before making extensive edits, verify:

1. **Purpose** — What is this document/code for?
2. **Audience** — Who reads or uses this?
3. **Context** — Internal vs external? Evergreen vs time-bound? Guidelines vs tracking?
4. **Objective** — What is the user actually trying to achieve?
5. **Constraints** — Spoken and unspoken rules, conventions, tone.

## Red Flags — Stop and Clarify

| Signal | Risk |
|--------|------|
| Adding time-sensitive content to evergreen docs | Dates become stale; misleads future readers |
| Adding project-specific details to general guidelines | Narrows audience; breaks generality |
| Mixing tracking/metrics with educational content | Conflates two document types |
| Uncertainty about the artifact's purpose | Any edit may be wrong direction |

## Common Pitfalls

| Mistake | Correct Approach |
|---------|-----------------|
| Start editing before reading the full document | Read to end first — later sections often clarify purpose |
| Apply "update X" by rewriting the whole thing | "Update X" = modify in place; preserve structure and tone |
| Assume the current document matches a standard template | Verify actual structure before conforming to templates |
| Ask for clarification after already starting edits | Ask before touching anything |
